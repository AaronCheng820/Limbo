34 # Introduction {#Introduction}

36 ## 1. Flex

42 * A Yacc-compatible parser generator that is part of the GNU Project.

44  Another famous related software is Yacc.

49  For details, please refer to [here](http://www.boost.org/doc/libs/1_55_0/libs/spirit/doc/html/index.html).

51 ## 4. LL Parser
52 
53 * In computer science, a LL parser is a top-down parser for a subset of context-free languages.
54  It parses the input from Left to right, performing Leftmost derivation of the sentence.
55 
56 ## 5. LR Parser
57 
58 * In computer science, LR parsers are a type of bottom-up parsers that efficiently handle
59  deterministic context-free languages in guaranteed linear time.
60 
61 ## 6. LL v.s. LR
62 
63 * Bison is based on LR parser and Boost.Spirit is based on LL parser.
64 Please refer to [here](http://cs.stackexchange.com/questions/43/language-theoretic-comparison-of-ll-and-lr-grammars).
65 
66 ## 7. LP and Min-Cost Flow
67 
68 * Min-cost flow (MCF) is a special case of linear programs (LP).
69 It is usually faster than general LP solver and is also able to achieve integer solution.
70 Some LP can be directly transformed to MCF problem, and for some other LP, its duality is a MCF problem.
71 In this library, an API for lemon (MFC solver) is implemented to solve LP and dual LP problems.
72 The API supports reading files with both LGF (lemon graph format) and LP (linear program format) and then dump out solutions.
73 Thus, it will be easier to verify results with general LP solvers such as Gurobi or CBC.

79 # Installation {#Installation}
80 
81 Some components depend on external libraries, such as
82 
83 * [Boost](www.boost.org): require BOOST_DIR environment variable to the path where Boost is installed. Limbo library is tested through Boost 1.55.0 to 1.57.0. BOOST_DIR should defined in a way that BOOST_DIR/include contains the headers to include and BOOST_DIR/lib contains the libraries to link.
84 * [Lemon](https://lemon.cs.elte.hu): require LEMON_DIR environment variable to the path where Lemon is installed.
85 * [Gurobi](www.gurobi.com): require GUROBI_HOME environment variable to the path where Gurobi is installed.
86 * [Flex](http://flex.sourceforge.net): require FLEX_DIR or LEX_INCLUDE_DIR environment variable if the flex version is not 2.5.37. See FAQ for details.
87 * [Zlib](http://www.zlib.net): require ZLIB_DIR, or ZLIB_INCLUDE_FLAG and ZLIB_LINK_FLAG environment variables to enable zlib features. If both zlib and boost are available, Gds parser supports reading/writing .gds.gz files. If using ZLIB_DIR, ZLIB_DIR should defined in a way that ZLIB_DIR/include contains the headers to include and ZLIB_DIR/lib contains the libraries to link; otherwise, use ZLIB_INCLUDE_FLAG and ZLIB_LINK_FLAG
88 
89 Users need to make sure they are properly installed and the corresponding settings are configured.
90 
91 ## 1. Default installation
92 
93 * In the directory of limbo library, run
94 ~~~~~~~~~~~~~~~~
95 make
96 make doc (optional) # generate documentation
97 make test (optional) # build test programs
98 make install PREFIX=an_absolute_path
99 ~~~~~~~~~~~~~~~~
100 
101 
102 The default installation path is the same as building path.
103 Please do not set PREFIX with a relative path.
104 After installation, it is strongly recommended to export LIMBO_DIR to the path where Limbo library is installed as an environment variable.
105 
106 ## 2. Customize CXX, CC and FC options
107 
108 * CXX sets the compiler for C++ and CC sets the compiler for C. FC sets the fortran compiler which is only needed by some third party packages like OpenBLAS.
109 Some examples are as follows,
110 ~~~~~~~~~~~~~~~~
111 make CXX=g++ CC=gcc FC=gfortran (default for Linux)
112 make CXX=clang++ CC=clang FC=gfortran (default for Mac/Darwin)
113 ~~~~~~~~~~~~~~~~
114 
115 
116 ## 3. Customize CXXSTDLIB and CXXSTD options
117 
118 * CXXSTDLIB is used to control -stdlib=xxx under clang++, and CXXSTD is used to control -std=xxx in most compilers.
119 
120 Under clang, following combinations are valid,
121 ~~~~~~~~~~~~~~~~
122 make CXXSTDLIB="-stdlib=libstdc++" CXXSTD="-std=c++98" (default)
123 make CXXSTDLIB="-stdlib=libc++" CXXSTD="-std=c++98"
124 make CXXSTDLIB="-stdlib=libc++" CXXSTD="-std=c++11"
125 ~~~~~~~~~~~~~~~~
126 
127 
128 Under gcc, there is no -stdlib option, so following combinations are valid,
129 ~~~~~~~~~~~~~~~~
130 make CXXSTD="-std=c++98" (default)
131 make CXXSTD="-std=c++11"
132 ~~~~~~~~~~~~~~~~
133 
134 
135 Users must make sure the setting is consistent to dependent libraries during compilation, such as Boost, Lemon, etc.
136 For example, if Boost is compiled with "clang++ -stdlib=libstdc++ -std=c++98", the same setting should be used for components dependent to Boost.
137 
138 ## 4. Customize OPENBLAS options
139 
140 * There is a third party OpenBLAS required by some other third party packages, such as Csdp. OPENBLAS option is used to control whether compiling these packages.
141 
142 If you do not need these packages, set OPENBLAS=0 (default);
143 otherwise, set OPENBLAS=1.
144 The default version of OpenBLAS is not very stable for cross platforms, which often results in compiling errors.
145 OpenBLAS is integrated as a submodule which fetch source code from remote repository, so it is always up-to-date version.
146 If you already have OpenBLAS in the directory, simply run "git submodule update" to fetch the latest version in the submodule directory.

170 ##3. Compiling errors related to LefScanner.cc usually come from the configurations of flex version and environment variables FLEX_DIR and LEX_INCLUDE_DIR.
171 
172 **A:** LefScanner.cc needs to include the correct FlexLexer.h from the flex package for compilation; i.e., the version of FlexLexer.h must match the version of the flex executable.
173 Most errors for LefScanner.cc are caused by the failure of finding the correct FlexLexer.h (be careful when you have multiple versions of flex installed).
174 To solve the problem, users can set the environment variable FLEX_DIR such that $FLEX_DIR/include points to the directory containing FlexLexer.h, or alternatively set LEX_INCLUDE_DIR to the directory containing FlexLexer.h.
175 The decision can be made according to how the flex package is installed.

178 ##4. Crappy linkage error under gcc 5.1 or later, even though libxxx.a is correctly linked, like
179 ~~~~~~~~~~~~~~~~
180 undefined reference to `GdsParser::read(GdsParser::GdsDataBaseKernel&, std::string const&)'
181 ~~~~~~~~~~~~~~~~
182 
183 **A:** The issue probably comes from potential inconsistent compiling configuration for Limbo and target program due to the new [Dual ABI](https://gcc.gnu.org/onlinedocs/libstdc%2B%2B/manual/using_dual_abi.html) introduced since gcc 5.1.
184 The compilation flag _GLIBCXX_USE_CXX11_ABI is used to control whether gcc uses new C++11 ABI, which is turned on in default.
185 If Limbo is compiled with C++11 ABI, while target program is compiled with old ABI, then the linkage error appears, vice versa.
186 Therefore, it is necessary to make sure the same STL ABI is used for compiling Limbo and target program.
187 In other words, set consistent _GLIBCXX_USE_CXX11_ABI values.
188 A safe way is to leave it to the default value.
